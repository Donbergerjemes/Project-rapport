01 — Introduction

Project Overview

▸ Executive Summary

The Hotel Management System is a fully server-rendered PHP web application designed to allow hotel administrators to manage the complete guest lifecycle — from booking creation through check-in, service assignment, payment, and feedback collection.

Project Goals

The system targets small-to-medium hotel operations and provides a centralized admin dashboard covering reservations, occupancy tracking, ancillary services, employee records, and guest satisfaction. The front-end delivers a responsive Bootstrap 5 interface with real-time price calculations driven by vanilla JavaScript XHR calls, while the back-end is a classical LAMP-stack (PHP + MariaDB).

02 — Design

System Architecture

The system follows a classic monolithic MVC-adjacent pattern where PHP files serve as both controllers and views. There is no framework — routing is handled by direct URL file access. All pages include db.php for the database connection and start a session to enforce admin authentication.

🌐
Presentation Layer

Bootstrap 5 HTML templates embedded in PHP. Three JS modules: scripts.js, ajax.js, login.js. SweetAlert2 for UX dialogs. Font Awesome icons. Custom styles.css.

↕
⚙️
Application Layer (PHP)

14 PHP files handling authentication, CRUD operations, session management, and business rules (double-booking prevention, occupancy limits, payment status). No MVC framework — files act as combined controller + view.

↕
🗄️
Data Layer (MariaDB)

8 tables: admins, clients, client_services, services, rooms, floors, staff, feedback. MySQLi extension with a mix of prepared statements and raw queries.

Request Flow

Every browser request goes directly to a PHP file. Pages that require authentication check $_SESSION['admin_id'] at the top and redirect to login.php if absent. AJAX calls (booking save, check-in/out, service payment) are handled by dedicated endpoint files and return plain-text tokens (Success, DoubleBooking, Error) consumed by JavaScript.

# Typical authenticated page request Browser ──GET──▶ dashboard.php │ include db.php (open MySQL connection) │ session_start() + check $_SESSION['admin_id'] │ Run SQL queries (bookings, stats, rooms) └──▶ Render HTML with embedded PHP data # AJAX booking save Browser ──POST──▶ book.php │ Validate POST fields │ Check double-booking (SQL SELECT) │ INSERT or UPDATE clients table └──▶ echo 'Success' | 'DoubleBooking' | 'Error'
03 — Components

Application Modules

🔐

Authentication

login.php · register.php · logout.php · login.js

SHA-256 password hashing for admin login. Register enforces a strong-password regex (uppercase, lowercase, digit, special char, min 8 chars). Logout destroys the session. login.js handles Enter-key navigation between fields.

📅

Booking Management

book.php · booking.php · ajax.js · scripts.js

Create and update guest reservations with room type, occupancy (adults/children), date range, and price. Double-booking is prevented server-side with an overlapping date query. Two near-duplicate PHP files exist (book.php / booking.php) that should be consolidated.

🏠

Check-in / Check-out

checkin.php · update_check_status.php

Two endpoints update the check_in_status enum. checkin.php also records actual_check_in_time and actual_check_out_time timestamps. Both are session-protected and use prepared statements.

💳

Payment & Invoicing

checkout.php · invoice.php · pay_service.php

checkout.php presents a card form that marks a booking as Paid. invoice.php renders a printable receipt and auto-triggers the browser print dialog. pay_service.php marks individual service charges as Paid.

🛎️

Services Module

services.php · pay_service.php

Hotel services (Breakfast, Laundry, Room Service, etc.) can be created and managed. Services are bookable at reservation time via checkboxes. Prices aggregate automatically in the front-end calculator, and service-level payment tracking is separate from the room charge.

👥

Staff Management

staff.php

Full CRUD for hotel employees with fields for name, position, email, phone, hire date, salary, and active/inactive status. Displays a staff roster table within the admin dashboard sidebar layout.

⭐

Guest Feedback

feedback.php

Admins can log 1–5 star ratings with comments linked to a guest record. Displays average rating and total review count. The feedback table is auto-created if absent (DDL on first page load). Feedback rows are deleted via a POST form with confirmation.

📊

Dashboard

dashboard.php

Central admin hub showing stat cards (total bookings, revenue, occupancy), a full bookings table with inline check-in/out and payment actions, and the booking creation/edit modal powered by JavaScript.

05 — Security Audit

Security Analysis

A review of all PHP files reveals a mixed security posture. Several endpoints use secure prepared statements, but critical booking endpoints rely on raw string interpolation, creating serious SQL injection vulnerabilities.

Critical
SQL Injection in book.php

Raw user input is directly interpolated into SQL strings (e.g., "SELECT id FROM clients WHERE room_number = $room_num AND check_in < '$out'"). An attacker can manipulate any booking query. Must be rewritten with $conn->prepare() and bound parameters.

Critical
No CSRF Protection on Sensitive Forms

Forms in checkout.php, feedback.php, and delete_booking.php perform state-changing actions (payment, deletion) without CSRF tokens. Any authenticated admin can be tricked into triggering these actions via a malicious third-party page.

Critical
checkout.php Has No Session Guard

checkout.php has no session_start() or $_SESSION['admin_id'] check. Any unauthenticated user can access any booking's payment page directly via ?id=N and mark it as Paid with a simple POST.

Warning
SHA-256 Used for Password Hashing (Not bcrypt)

login.php and register.php use hash('sha256', $password). SHA-256 is a fast hash — it allows rapid brute-force attacks. PHP's password_hash($pass, PASSWORD_BCRYPT) with password_verify() should be used instead.

Warning
Hardcoded MySQL Root Credentials in db.php

db.php connects as root with an empty password. A dedicated least-privilege database user should be created, and credentials should be stored in environment variables or a config file outside the web root.

Warning
invoice.php Has No Authentication Guard

The invoice page fetches and displays full guest data (name, email, price) for any ?id=N without checking if a session exists. Any unauthenticated visitor can enumerate booking invoices.

Notice
Duplicate Booking Handlers (book.php vs booking.php)

Two nearly identical files handle booking creation/updates. book.php uses raw queries (vulnerable); booking.php uses the same raw queries but with different validation messages (French comments). These should be merged into one secure, prepared-statement-based file.

Notice
feedback.php Creates Tables at Runtime

The feedback page runs CREATE TABLE IF NOT EXISTS and ALTER TABLE on every page load. This is a development convenience that should be replaced with proper database migration scripts in production.

Good
Prepared Statements Used in Auth, Delete, Check-in

login.php, register.php, delete_booking.php, checkin.php, update_check_status.php, pay_service.php, and feedback.php all correctly use $conn->prepare() with bound parameters.

Good
Output Escaping with htmlspecialchars()

Displayed user-supplied values use htmlspecialchars() consistently across templates, preventing reflected XSS. The admin name in the sidebar and all client fields are properly escaped.

Good
Strong Password Policy on Registration

register.php enforces a regex requiring uppercase, lowercase, digit, special character, and minimum 8 characters. This is a solid baseline for admin account creation.

06 — Action Plan

Recommendations

High Priority
01

Rewrite book.php with Prepared Statements

Replace all string-interpolated SQL in book.php and booking.php with $conn->prepare(). Merge both files into a single, secure endpoint.

High Priority
02

Add Session Guards to checkout.php & invoice.php

Both pages must call session_start() and verify $_SESSION['admin_id'] before serving any content or processing POST data.

High Priority
03

Implement CSRF Tokens

Generate a per-session CSRF token, embed it in all forms as a hidden field, and validate it server-side before processing any state-changing request.

High Priority
04

Switch to password_hash / password_verify

Replace hash('sha256', …) with PHP's native password_hash($pass, PASSWORD_BCRYPT) and password_verify() for all admin password operations.

Medium Priority
05

Move Credentials to Environment Variables

Store MySQL credentials in a .env file (loaded via phpdotenv) or server environment variables. Create a dedicated least-privilege DB user instead of using root.

Medium Priority
06

Populate and Use the rooms Table

The rooms table exists in the schema but contains no data. Room availability checks rely on the clients table alone. Populating rooms enables proper room inventory management.

Medium Priority
07

Move DDL Out of Application Code

Remove CREATE TABLE IF NOT EXISTS and ALTER TABLE from feedback.php. Use versioned database migration scripts (e.g., Phinx) instead.

Low Priority
08

Implement Real Payment Processing

checkout.php collects card data but performs no real processing. Integrate a gateway (Stripe, PayPal) or remove the form entirely and use an external payment link flow.

Low Priority
09

Add Input Validation for Email Fields

The database contains entries like kakagmail.com and kljıof3rn.cm — invalid emails accepted without server-side validation. Use PHP's filter_var($email, FILTER_VALIDATE_EMAIL).
